# MadenPlugin

MadenPlugin is an open-source Minecraft plugin built for servers that need a managed custom mine system instead of a simple vanilla block-breaking flow. The plugin allows you to define one or more mine regions, decide which block types inside those regions are handled by the plugin, give custom reward items when players break those blocks, and automatically restore the mine after the configured respawn time.

This project was designed mainly for Skyblock-style servers, but it can also be used on survival, economy, prison-lite or custom progression servers where mine blocks need to produce special rewards instead of their default drops.

The plugin does not globally change block breaking across the server. It only manages blocks that meet all of the following conditions:

- The block is inside a configured mine region.
- The block type is registered for that mine.
- The mine area was scanned with the `/maden tara` command.
- The specific block was recorded as a tracked mine block.

That design keeps the system controlled and predictable. Normal blocks outside configured mines are not touched.

## What The Plugin Does

At a high level, MadenPlugin works like this:

1. You create a mine with a name.
2. You select the mine you want to edit.
3. You mark the first and second corners of the mine region.
4. You define which Minecraft block types belong to that mine.
5. You scan the mine area so the plugin records the matching blocks.
6. Players break those tracked blocks.
7. The plugin gives configured reward items instead of normal mine progression.
8. After the configured respawn time, the mine fills back in.

This means the plugin is not just a "reset mine" plugin. It is also a custom reward system, a progression system, and an economy integration point.

## Main Features

- Multiple named mines
- Per-mine region selection
- Per-mine tracked block type definitions
- Custom reward item material
- Custom reward item amount
- Custom reward item display name
- Custom reward item lore
- Automatic full mine reset after respawn time
- Optional break message per mine
- Optional break sound per mine
- Optional console commands per mine
- SQLite persistence for tracked mine blocks
- PlaceholderAPI support
- Compatibility-oriented release target for Minecraft `1.16.5+`
- Scoped managed-break behavior for tracked mine blocks

## Intended Use Cases

MadenPlugin fits especially well in these cases:

- Lightweight prison-style mine regions without a heavy full prison stack

## Installation

1. Take the release jar from [`artifacts/MadenPlugin-2.3.0.jar`](./artifacts/MadenPlugin-2.3.0.jar).
2. Copy it into your server's `plugins` folder.
3. Start or restart the server.
4. Let the plugin generate its folder and default files.
5. Use the in-game commands to create and configure your first mine.

If you previously used older MadenPlugin jars, remove the old ones first so only one active jar remains in your `plugins` folder.

## Files Generated By The Plugin

After first startup, the plugin uses these main files:

- `plugin.yml`
- `config.yml`
- `blocks.yml`
- `tracked-blocks.db`

The most important files for normal setup are:

- `config.yml`
  This stores mine definitions such as region positions, respawn time, optional messages, optional sound settings and optional commands.

- `blocks.yml`
  This stores block type definitions for each mine, including the source block and reward item details.

- `tracked-blocks.db`
  This is the SQLite database that stores scanned and tracked mine blocks.

## Command Reference In Detail

### `/maden madenekle <isim>`

Creates a new mine and selects it for the player who ran the command.

Example:

```text
/maden madenekle obsimaden
```

What happens:

- A mine named `obsimaden` is created.
- The player's current world is stored as the mine world.
- The player's current location is used as the initial `pos1` and `pos2`.
- The mine gets default settings such as a 60 second respawn time.
- The mine becomes the currently selected mine for that admin.

### `/maden sec <isim>`

Selects an existing mine so the next edit commands affect that mine.

Example:

```text
/maden sec obsimaden
```

You should use this when:

- You have multiple mines.
- You want to switch from editing one mine to another.

### `/maden pos1`

Sets the first corner of the selected mine to your current location.

### `/maden pos2`

Sets the second corner of the selected mine to your current location.

Together, `pos1` and `pos2` define the rectangular 3D area of the mine.

Important:

- The mine area includes all blocks between those two points.
- Both points should be in the same world.
- The plugin scans blocks inside that full region.

### `/maden blokekle <minecraft_blok_adi> <isim>`

Registers a block type for the selected mine.

Example:

```text
/maden blokekle OBSIDIAN Obsidyen Parcasi
```

What this means:

- `OBSIDIAN` is the source block inside the mine.
- `Obsidyen Parcasi` is the display name of the reward item.

The source block name must be a valid Minecraft material name.

### `/maden bloksil <minecraft_blok_adi>`

Removes a block type definition from the selected mine.

Example:

```text
/maden bloksil OBSIDIAN
```

Use this when:

- You no longer want that source block to be part of the mine.
- You want to scan again after changing the mine content.

### `/maden blokduzenle <minecraft_blok_adi> <yeni_isim>`

Changes the display name of the reward item for the selected block type.

Example:

```text
/maden blokduzenle OBSIDIAN Saf Obsidyen Parcasi
```

### `/maden bloklist`

Lists the block types currently defined for the selected mine.

This is useful when:

- You forgot which blocks are configured.
- You are checking whether the right block types were added before scanning.

### `/maden tara`

Scans the selected mine region and records every matching configured block inside it.

This is one of the most important commands in the plugin.

Without scanning:

- The region exists.
- The block types may exist.
- But the mine is not yet actively tracking the actual blocks in the world.

After scanning:

- Matching blocks inside the region are registered.
- Those specific blocks become managed mine blocks.
- Breaking them will trigger reward and respawn logic.

### `/maden bilgi`

Shows information about the selected mine.

Typical data shown:

- Selected mine name
- Total mine count
- Block type count
- Total tracked block count
- Global time placeholder value
- Mine-specific time value
- Mine fill percentage
- Current scan state

### `/maden reload`

Reloads plugin data from disk.

Use this after:

- Editing configuration files manually
- Changing mine settings
- Changing block reward definitions

### `/maden madensil <isim>`

Deletes the selected mine definition and removes its tracked block data.

Be careful with this command. It is intended for cleanup or rebuilding a mine from scratch.

## First-Time Setup Guide

This section explains exactly how to create your first mine from zero.

### Step 1: Install the plugin

Put the jar in the `plugins` folder and start the server.

### Step 2: Decide the mine area

Before using commands, choose where the mine should exist in the world. Stand at one corner of the area you want to use.

### Step 3: Create the mine

```text
/maden madenekle obsimaden
```

Now the mine exists and is selected.

### Step 4: Set the first point

Stand at the first corner and run:

```text
/maden pos1
```

### Step 5: Set the second point

Move to the opposite corner and run:

```text
/maden pos2
```

At this point, the plugin knows the region size of the mine.

### Step 6: Add block types

If your mine contains obsidian and diamond ore, register them:

```text
/maden blokekle OBSIDIAN Obsidyen Parcasi
/maden blokekle DIAMOND_ORE Elmas Parcasi
```

This does not scan the mine yet. It only tells the plugin which block types count as managed mine blocks.

### Step 7: Scan the mine

```text
/maden tara
```

The plugin now checks the entire selected region and records blocks that match the block types you added.

### Step 8: Test the mine

Break one of the scanned blocks as a normal survival player.

Expected result:

- The block is handled by the plugin.
- Default drop behavior is suppressed.
- The configured reward item is given.
- The mine starts or continues its respawn timing.

## Example Setup Walkthrough

Here is a complete practical example.

Assume you want a mine called `kristalmaden`.

Commands:

```text
/maden madenekle kristalmaden
/maden sec kristalmaden
/maden pos1
/maden pos2
/maden blokekle OBSIDIAN Kristal Parca
/maden blokekle EMERALD_ORE Zenginlik Parca
/maden blokekle GOLD_ORE Altin Parca
/maden tara
```

What this setup does:

- Creates a mine named `kristalmaden`
- Defines the mine region using your two points
- Registers obsidian, emerald ore and gold ore as managed mine block types
- Scans the region
- Records all matching blocks inside the selected area

When players mine those tracked blocks:

- They receive the configured custom reward items
- The original block does not use normal drop logic
- The plugin tracks that the block is currently broken
- The mine refills after the configured respawn time

## How Block Tracking Works

This part is important for understanding why scanning matters.

The plugin does not automatically control every matching material in the entire world. It controls only tracked mine blocks.

Tracked mine blocks are created by scanning.

That means:

- If a block is in the mine region but the mine was never scanned, it is not active.
- If a block material was not registered with `blokekle`, it is ignored during scanning.
- If a block exists outside the mine region, it is ignored.

This makes the system predictable and keeps the plugin from affecting unrelated gameplay.

## How Rewards Work

Each configured block type has reward data.

Reward data includes:

- Reward material
- Reward amount
- Reward display name
- Reward lore

By default, when you use `/maden blokekle`, the plugin creates a reward item using the same material as the source block and assigns the display name you gave in the command.

Later, if needed, you can adjust reward-related details directly in `blocks.yml`.

## `blocks.yml` Explained

`blocks.yml` stores the per-mine block definitions.

Example structure:

```yaml
block-types:
  obsimaden:
    obsidian:
      source: OBSIDIAN
      reward:
        material: OBSIDIAN
        amount: 1
        name: "&5Obsidyen Parcasi"
        lore:
          - "&7Madenden elde edildi."
```

Meaning:

- `obsimaden` is the mine id
- `obsidian` is the block type id
- `source` is the real block inside the mine
- `reward.material` is the item material given to the player
- `reward.amount` is the item amount
- `reward.name` is the visible item name
- `reward.lore` is the item lore

After editing `blocks.yml`, use:

```text
/maden reload
```

If you changed actual mine contents or block types significantly, scan again afterward.

## `config.yml` Explained

`config.yml` stores mine-level settings.

Important parts:

- World name
- `pos1`
- `pos2`
- `respawn-seconds`
- optional message settings
- optional sound settings
- optional command settings

Example structure:

```yaml
messages:
  mine-ready: "&aHazir"

mines:
  obsimaden:
    world: world
    pos1:
      world: world
      x: 0
      y: 64
      z: 0
    pos2:
      world: world
      x: 10
      y: 70
      z: 10
    respawn-seconds: 60
    messages:
      enabled: true
      break: "&aMadenden &f%item% &aelde ettin."
    sound:
      enabled: true
      name: "ENTITY_EXPERIENCE_ORB_PICKUP"
      volume: 1.0
      pitch: 1.0
    commands:
      enabled: false
      list:
        - "say %player_name% maden odulu kazandi"
```

## Mine Respawn Logic

The plugin restores the mine after the configured respawn time.

How it behaves:

- A tracked block is broken.
- The block becomes air.
- The mine starts or continues its reset timer.
- When the timer completes, the tracked blocks for that mine are restored to their original configured source materials.

This is a mine-level reset flow, not random per-player private block handling.

## Optional Effects

Each mine can optionally do extra things when a tracked block is broken.

### Break Message

If enabled, the player receives a message when they break a managed mine block.

Supported placeholders inside the break message:

- `%player%`
- `%item%`

### Break Sound

If enabled, the player hears the configured sound.

### Console Commands

If enabled, the plugin executes the listed commands from console when a tracked block is broken.

Useful placeholders:

- `%player%`
- `%player_name%`

This makes the plugin easy to integrate with other economy or reward plugins.

## PlaceholderAPI Support

The plugin automatically registers its placeholders if PlaceholderAPI is installed and enabled.

Available placeholders:

- `%maden_time%`
- `%maden_time_<madenadi>%`
- `%maden_percent_<madenadi>%`

### `%maden_time%`

Shows the nearest active remaining reset time across all mines.

If no mine is currently waiting for reset, it shows the configured ready message.

### `%maden_time_<madenadi>%`

Shows the remaining time for a specific mine.

Example:

```text
%maden_time_obsimaden%
```

### `%maden_percent_<madenadi>%`

Shows the fill percentage of a specific mine based on tracked block state.

Example:

```text
%maden_percent_obsimaden%
```

## WorldGuard And Protection Plugin Behavior

The plugin is built to manage only selected mine blocks in a controlled way.

This means the managed break flow is intentionally scoped:

- It targets tracked mine blocks only
- It does not turn the whole server into an unrestricted mining area
- It does not affect random protected blocks outside the mine system

In practice, the plugin is meant to make configured mine blocks work as mine blocks, while leaving unrelated blocks alone.

## Performance Notes

The mine scan is handled in batches rather than trying to process the entire region in one single heavy operation.

This helps reduce lag spikes, especially on larger mine regions.

Later runtime operations are also kept focused on tracked blocks only, which helps avoid unnecessary world-wide checks.

SQLite is used for persistence so the plugin can recover tracked mine state after restart more safely than a pure YAML-only runtime tracking model.

## Persistence

Tracked mine blocks are stored in SQLite.

This gives the following benefits:

- Mine state survives restart
- Broken tracked blocks can be restored correctly after reload
- Tracked mine records stay organized
- Large tracked mine sets are easier to manage safely

## Typical Mistakes And How To Avoid Them

### Mistake 1: Creating a mine but forgetting to scan

If you create the mine and set positions but never run `/maden tara`, the plugin has no tracked blocks yet.

Fix:

```text
/maden tara
```

### Mistake 2: Scanning before adding block types

If you scan first and only add block types later, the earlier scan did not know what to track.

Fix:

1. Add block types
2. Scan again

### Mistake 3: Editing the wrong mine

If you have multiple mines and forget to switch selection, your changes may go to the wrong mine.

Fix:

```text
/maden sec <isim>
```

### Mistake 4: Using invalid Minecraft material names

`blokekle` requires a valid Minecraft material name.

Fix:

Use official material names such as:

- `OBSIDIAN`
- `DIAMOND_ORE`
- `EMERALD_ORE`
- `STONE`

### Mistake 5: Changing the region but not rescanning

If you move `pos1` or `pos2`, the tracked mine records do not automatically rebuild themselves.

Fix:

Run:

```text
/maden tara
```

after region changes.

1. Install the jar

3. Create the mine
4. Select the mine
5. Set `pos1`
